# IERC8183 接口规范文档

> 项目：Trustless Agent Work Agreements
> 协议：ERC-8183 Agentic Commerce Protocol
> 文件：contracts/interfaces/IERC8183.sol
> 版本：Draft，待审阅

---

## 1. 概述

IERC8183 是 ERC-8183 标准的核心接口，定义了 Agentic Commerce（自动化商业）的链上托管结算协议。

**一句话**：Client 锁定资金 → Provider 提交工作 → Evaluator 裁决放款/退款。

### 1.1 继承关系

```
IERC165（接口检测）
    ↑
IERC8183（托管结算主接口）
    ↑
ERC8183Escrow（本项目的实现合约）
```

继承 IERC165 使外部可调用 `supportsInterface(接口ID)` 识别合约是否为 ERC-8183 实现。

### 1.2 RFC 2119 关键词约定

| 关键词 | 含义 |
|--------|------|
| MUST / SHALL | 必须实现，违反即为不合规 |
| MUST NOT / SHALL NOT | 绝对禁止 |
| SHOULD | 推荐，特殊情况下可偏离 |
| MAY / OPTIONAL | 可选，实现方可自行决定 |

## 2. 状态机

### 2.1 枚举定义

| 值 | 名称 | 类型 | 含义 |
|:--:|------|------|------|
| 0 | `Open` | 中间态 | 已创建，可设 budget，可 fund 或 reject |
| 1 | `Funded` | 中间态 | 资金已托管；Provider 可 submit，Evaluator 可 reject |
| 2 | `Submitted` | 中间态 | Provider 已提交；仅 Evaluator 可 complete 或 reject |
| 3 | `Completed` | 终态 | 放款给 Provider（扣除平台费 + Evaluator 费） |
| 4 | `Rejected` | 终态 | 退款给 Client |
| 5 | `Expired` | 终态 | 超时退款，同 Rejected |

### 2.2 合法转换

```
Open ──fund()──→ Funded ──submit()──→ Submitted ──complete()──→ Completed
  │                │                      │
  │                │                      ├──reject()──→ Rejected
  │                │                      └──claimRefund()──→ Expired
  │                ├──reject()──→ Rejected
  │                └──claimRefund()──→ Expired
  └──reject()──→ Rejected
```

终端态不可逆。

## 3. 角色与权限矩阵

| 操作 | 前提状态 | Client | Provider | Evaluator | Anyone |
|------|:--------:|:------:|:--------:|:---------:|:------:|
| `createJob` | — | ✅ | — | — | — |
| `setProvider` | Open | ✅ | — | — | — |
| `setBudget` | Open | ✅ | ✅ | — | — |
| `fund` | Open | ✅ | — | — | — |
| `submit` | Funded | — | ✅ | — | — |
| `complete` | Submitted | — | — | ✅ | — |
| `reject` | Open | ✅ | — | — | — |
| `reject` | Funded | — | — | ✅ | — |
| `reject` | Submitted | — | — | ✅ | — |
| `claimRefund` | Funded/Submitted | — | — | — | ✅ |

**关键设计决策**：

1. Funded 后 Client 不能 reject — Provider 开工后的权益保护。Client 唯一退路是超时 `expiredAt`
2. Submitted 后只有 Evaluator — 裁决权交给第三方
3. Client = Evaluator 合法 — 不需要第三方不强制引入
4. claimRefund 任何人都能调 — 防止 Client 失联导致资金锁死
5. **evaluator 创建时确定且不可变** — EIP 原文 `set at creation, MUST be set`，不提供 setEvaluator 函数。防止 Client 中途换成自己控制的 Evaluator 损害 Provider 权益。Evaluator 掉线时靠 expiredAt → claimRefund → 重开新 Job

## 4. 数据结构

### 4.1 Job Struct

```solidity
struct Job {
    address client;
    address provider;    // 可为 address(0)，后补
    address evaluator;   // 创建时确定，不可变更
    string  description;
    uint256 budget;
    uint256 expiredAt;
    Status  status;
    address hook;        // address(0) = 不启用
}
```

| 字段 | 可变更 | 说明 |
|------|:---:|------|
| `client` | ❌ | `createJob` 的 msg.sender |
| `provider` | ✅ | `setProvider` 后补/更换（仅 Open） |
| `evaluator` | ❌ | EIP 规定 `set at creation`，无 set 函数 |
| `budget` | ✅ | `setBudget` 多次议价（仅 Open） |
| `hook` | ❌ | 创建时绑定 |

### 4.2 支付代币

`paymentToken()` 返回合约接受的 ERC-20 地址，全局统一（非 per-job）。

## 5. 事件设计

EIP-8183 要求至少 10 个事件：

### 状态事件

| 事件 | 触发函数 | indexed 参数 | 含义 |
|------|----------|-------------|------|
| `JobCreated` | createJob | jobId, client, provider | 新任务 |
| `ProviderSet` | setProvider | jobId, provider | 接单方确定 |
| `BudgetSet` | setBudget | jobId | 预算设定（`amount` 参数） |
| `JobFunded` | fund | jobId, client | 资金锁定 |
| `JobSubmitted` | submit | jobId, provider | 交付物提交 |
| `JobCompleted` | complete | jobId, evaluator | 放款确认 |
| `JobRejected` | reject | jobId, rejector | 拒绝（rejector = Client 或 Evaluator） |
| `JobExpired` | claimRefund | jobId | 超时退款 |

### 资金事件

| 事件 | 触发函数 | indexed 参数 | 含义 |
|------|----------|-------------|------|
| `PaymentReleased` | complete | jobId, provider | 放款金额明细 |
| `Refunded` | reject / claimRefund | jobId, client | 退款金额明细 |

## 6. Fees（费用模型）

EIP-8183 的收费是 **OPTIONAL（MAY）**——合约可以选择不收。

### 收费规则

| 规则 | 说明 |
|------|------|
| 计费方式 | basis points（bps，万分比）。1 bps = 0.01%，10000 bps = 100% |
| 收费时机 | **仅在 Completed 时**扣除。Rejected / Expired 不收费 |
| 收费对象 | 从 Provider 的收款中扣除（`payout = budget - platformFee - evaluatorFee`） |
| 收款方 | 可配置的 treasury（财库）地址和 evaluator 地址 |
| 强制上限 | `platformFee + evaluatorFee ≤ 10000`（≤ 100%） |

### 为什么只在 Completed 时收费

交易没成（reject / expire），钱全额退给 Client。平台不赚中介费——公平设计。

### 我们项目的处理

- MVP 阶段 **暂不设置平台费**（`platformFeeBP = 0`）
- 保留 `evaluatorFeeBP` 字段在合约中，方便后续配置
- Demo 展示时资金流向清晰：Client → Escrow → Provider（无中间抽成）

> 对应 EIP 原文：`Implementations MAY charge a platform fee (basis points) on Completed, paid to a configurable treasury.`

## 7. Core Functions

### 7.1 createJob

```
createJob(provider, evaluator, expiredAt, description, hook) → jobId
```

| | 内容 |
|---|---|
| **调用者** | Client（`client = msg.sender`） |
| **Revert 条件** | `evaluator == address(0)`；`expiredAt ≤ block.timestamp` |
| **行为** | 创建 Job，状态 Open；provider 可为 address(0)；hook 可为 address(0) |
| **optParams** | 不适用（无 Hook 参与创建） |

### 7.2 setProvider

```
setProvider(jobId, provider)                  // 无 optParams
setProvider(jobId, provider, optParams)        // 有 optParams（OPTIONAL）
```

| | 内容 |
|---|---|
| **调用者** | Client only |
| **Revert 条件** | `status ≠ Open`；当前 `job.provider ≠ address(0)`；`provider == address(0)` |
| **行为** | 设置 `job.provider = provider`，emit `ProviderSet` |
| **optParams** | OPTIONAL，透传给 Hook |

### 7.3 setBudget

```
setBudget(jobId, amount)
setBudget(jobId, amount, optParams)
```

| | 内容 |
|---|---|
| **调用者** | Client 或 Provider（双方议价） |
| **Revert 条件** | `status ≠ Open`；`msg.sender ≠ client 且 ≠ provider` |
| **行为** | 设置 `job.budget = amount`，emit `BudgetSet`。可多次调用——最后一次值为准 |
| **optParams** | OPTIONAL，透传给 Hook |

### 7.4 fund

```
fund(jobId, expectedBudget)
fund(jobId, expectedBudget, optParams)
```

| | 内容 |
|---|---|
| **调用者** | Client only |
| **Revert 条件** | `status ≠ Open`；`msg.sender ≠ client`；`budget == 0`；`provider == address(0)`（provider 未设置）；`job.budget ≠ expectedBudget`（防 front-run） |
| **行为** | `token.transferFrom(client, escrow, budget)` 扣款；`status = Funded`；emit `JobFunded` |
| **optParams** | OPTIONAL，透传给 Hook |

> **expectedBudget 安全意义**：Alice 设 budget=100，Malory 插队改为 1000。fund 时 `expectedBudget(100) ≠ 实际 budget(1000)` → revert，Alice 无损。

### 7.5 submit

```
submit(jobId, deliverable)
submit(jobId, deliverable, optParams)
```

| | 内容 |
|---|---|
| **调用者** | Provider only |
| **Revert 条件** | `status ≠ Funded`；`msg.sender ≠ job.provider` |
| **行为** | `status = Submitted`；emit `JobSubmitted(jobId, provider, deliverable)` |
| **deliverable** | `bytes32`，链下交付物哈希（Keccak-256 / IPFS CID）。链上不验证内容 |
| **optParams** | OPTIONAL，透传给 Hook |

### 7.6 complete

```
complete(jobId, reason)
complete(jobId, reason, optParams)
```

| | 内容 |
|---|---|
| **调用者** | Evaluator only |
| **Revert 条件** | `status ≠ Submitted`；`msg.sender ≠ job.evaluator` |
| **行为** | `status = Completed`；转账 `provider 收款 = budget - platformFee - evaluatorFee`；emit `JobCompleted` + `PaymentReleased` |
| **optParams** | OPTIONAL，透传给 Hook |

### 7.7 reject

```
reject(jobId, reason)
reject(jobId, reason, optParams)
```

| | 内容 |
|---|---|
| **调用者** | Client（Open 时）或 Evaluator（Funded / Submitted 时） |
| **Revert 条件** | `status 不是 Open/Funded/Submitted`；`msg.sender ≠ client`（Open 时）且 `≠ evaluator`（Funded/Submitted 时） |
| **行为** | `status = Rejected`；若 Funded 或 Submitted 则退款给 Client；emit `JobRejected(jobId, rejector, reason)` + `Refunded` |
| **optParams** | OPTIONAL，透传给 Hook |

### 7.8 claimRefund

```
claimRefund(jobId)
```

| | 内容 |
|---|---|
| **调用者** | **任何人**（RECOMMENDED）或 Client only（MAY） |
| **Revert 条件** | `status 不是 Funded 或 Submitted`；`block.timestamp < expiredAt`（未过期） |
| **行为** | `status = Expired`；全额退款给 Client；emit `JobExpired` + `Refunded` |
| **optParams** | **无**。单版本、不可 Hook——保证退款通路永不锁死 |

## 8. Attestation（裁决证明）

`complete()` 和 `reject()` 的 `reason` 参数是实现 attestation 的链上锚点——Evaluator 不只是说「完成/不完成」，而是出具一份可查证的证明。

### reason 参数

- 类型：`bytes32`，OPTIONAL（可为 `bytes32(0)`）
- 典型用途：链下评估报告的 Keccak-256 哈希
- 可被 ERC-8004 声誉系统读取，作为声誉记录的判断依据

### 我们项目中的使用

LLM Evaluator 评分后：
1. 生成验证报告（checklist 5 项 × 通过/不通过 + 理由）
2. `reason = keccak256(报告全文)`
3. 调用 `complete(jobId, reason)` 或 `reject(jobId, reason)`
4. 将来任何人重算 keccak256 即可验证 Evaluator 当时确实评审过此报告

## 9. Hook 扩展机制

### 9.1 接口

```solidity
interface IACPHook {
    function beforeAction(uint256 jobId, bytes4 selector, bytes calldata data) external;
    function afterAction(uint256 jobId, bytes4 selector, bytes calldata data) external;
}
```

### 9.2 可 Hook 函数与 optParams

optParams 是 OPTIONAL——不传时调无参版本，传给 Hook 时调有参版本。合约不解码 optParams，直接透传。

| 函数 | optParams 典型用途 |
|------|-------------------|
| setProvider | BiddingHook：签名验证 |
| setBudget | BiddingHook：竞标价 |
| fund | FundTransferHook：approve 验证 |
| submit | FundTransferHook：pull 产出代币 |
| complete | ERC-8004 声誉写入 |
| reject | 状态清理 |

### 9.3 claimRefund 不可 Hook

**唯一不可 Hook 的函数**。如果 Hook 有权拦截 claimRefund，恶意 Hook 可永远锁死资金。这是协议级安全保证。

## 10. 与 CAW 的集成边界

| 层 | 职责 | 工具 |
|----|------|------|
| 接口定义 | 函数签名 + 状态机 + 事件 | IERC8183.sol |
| 托管结算 | 资金锁定 + 状态流转 + 放款 | ERC8183Escrow.sol |
| 权限控制 | Pact 三层约束 + 签名前校验 | CAW Policy 引擎 |
| Agent 操作 | 构造 Pact → 调合约 | CAW CLI + Python 脚本 |

协作线：CAW Pact 白名单指定合约地址 + 函数选择器（fund → submit → complete），越界调用在 Policy 引擎层拒绝。

## 11. 已知局限

| 局限 | 影响 | 缓解 |
|------|------|------|
| Evaluator 单点信任 | Submitted 后 Evaluator 可任意决定 | 多签 / zkProof（未来） |
| 无争议仲裁 | 只有 complete/reject 二元 | 协议刻意简洁 |
| 交付物仅哈希 | 链上不验证 | Evaluator 链下验证 |
| 单链 | 不支持跨链 Job | ERC-7683（未来） |
| 不可升级 | 固定逻辑 | UUPS/透明代理（可选） |
